# Technical Design Specification: Scaling High-Precision AI Cognition via Montague Semantics

## 1. Architectural Pivot: From Pragmatic Extraction to Formal Compositionality

The transition from the "Local Acquirer" Minimum Viable Product (MVP) to a robust cognitive system necessitates a strategic pivot from heuristic-based triplet extraction to a higher-order semantic framework. While the MVP utilized Large Language Models (LLMs) to identify "flat" (Entity-Relationship-Entity) structures, autonomous synthesis requires a system capable of interpreting the "logic of ordinary English" with mathematical rigor. We must move beyond the probabilistic "best guess" of extraction prompts toward a formal compositional engine that treats natural language as a formal system.

### System Architecture Overview

The following table reflects the complete stack, moving from the ingestion abstraction down to the conversational output:

| Component | Architecture Role | Functional Mechanism |
| :--- | :--- | :--- |
| **Data Ingestion** | Abstract Source Classes | Normalizes unstructured inputs (PDF, Web, API) into clean text streams. |
| **Semantic Engine** | Montague / $\lambda$-calculus | Converts text to formally typed mathematical representations. |
| **World Model ($M$)** | Graph Database | Stores extensional data where $M, g \models \phi$ is verified. |
| **Synthesis Engine** | Internal Zettelkasten | Evaluates intensional logic and generates higher-order insight nodes. |
| **Server Protocol** | MCP Coordinator | Exposes the Graph and Zettelkasten as standard tools to AI clients. |
| **Interface** | Conversational Loop | Facilitates human-guided hypothesis testing and model steering. |

### The Direct Compositionality Hypothesis

We adopt the hypothesis of Direct Compositionality, adhering to the slogan that "syntax and semantics work in tandem." Unlike systems that postpone interpretation to a "Logical Form" (LF) level—typical of Quantifier Raising (QR) paradigms—our engine interprets constituents as they are formed.

We distinguish between Strong Direct Compositionality (where syntactic operations are restricted to simple concatenation with no reference to internal structure) and Weak Direct Compositionality (allowing for "Wrap" or "Infixation" operations, as defined by Bach and Dowty). While Strong Compositionality offers the highest degree of "compositional transparency," our engine utilizes Weak Direct Compositionality to handle the "Wrap" of complex-transitive verbs around their objects. This allows the system to provide a precise semantic value $\llbracket \alpha \rrbracket^g$ for a constituent like "saw everyone" before it is embedded in a larger sentential structure, ensuring high-precision synthesis of disparate data points.

## 2. Integrated Data Engineering Backbone

For an autonomous system to ingest diverse data over decades, it requires a robust preprocessing layer rather than jumping straight from raw text to logical parsing. We implement an abstract data engineering backbone to standardize the ingestion pipeline. This pipeline handles data from any source and ensures it is consistently processed before reaching the Montague engine. We define concrete subclasses—such as `PDFSource`, `WebScraperSource`, and `APISource`—to normalize inputs into clean text streams. Adding this layer guarantees the scalability and maintainability necessary for feeding the AI a continuous, clean stream of information.

## 3. Formal Foundations: The Montague Semantic Engine

The core of our system is a Model-Theoretic Semantic engine. This provides the "truth-conditional" anchor required to ensure that autonomous synthesis is logically grounded rather than merely hallucinatory. In this framework, the Neo4j graph database serves as our World Model $M$. Truth is not an intuition; it is a formal relation $M, g \models \phi$, where the database satisfies a logical formula under a specific assignment of variables.

### The Principle of Compositionality

The engine implements the Principle of Compositionality: the meaning of a complex Zettel (insight) is a function of the meanings of its parts (atomic facts) and their mode of syntactic combination. Formally, for any complex expression $\gamma$ formed from $\alpha$ and $\beta$, the denotation is $\llbracket \gamma \rrbracket = \mathcal{F}(\llbracket \alpha \rrbracket, \llbracket \beta \rrbracket)$. This ensures that every high-level pattern generated by the synthesis engine is traceably derived from verified atomic nodes in the graph.

### Type-Driven Interpretation via $\lambda$-Abstraction

The engine employs Lambda ($\lambda$) Calculus to translate natural language inputs into higher-order functions. These abstractions are then mapped directly to physical storage operations:

* **Property Lifting**: A proper name like "Anvil Corp" is lifted to a set of properties: $\lambda P[P(\text{Anvil Corp})]$. In Neo4j, this translates to a node lookup where the name property equals "Anvil Corp."
* **Functional Application**: A verb like "is based in" is treated as a function $\lambda x \lambda y [\text{BasedIn}(y, x)]$.
* **Storage Mapping**: The result of a $\lambda$-conversion (e.g., $\text{BasedIn}(\text{Anvil Corp}, \text{Anaconda})$) is instantiated as a directed relationship `(:Entity {name: 'Anvil Corp'})-[:IS_BASED_IN]->(:Entity {name: 'Anaconda'})`.

### Generalized Quantifiers and Entailment

To manage complex data ingestion, we utilize Generalized Quantifiers (e.g., "every," "most," "no"). This allows the system to characterize Downward Entailing contexts, which is essential for negative polarity licensing. For example, the quantifier "no" licenses an inference from a superset ("companies") to a subset ("logistics companies"). If the system ingests "No company in Anaconda is insolvent," it can logically entail "No logistics company in Anaconda is insolvent" without requiring a separate data entry.

## 4. Formalizing the Neuro-Symbolic Bridge

While the destination is formal truth-conditional representations, the probabilistic Large Language Model must reliably output these complex $\lambda$-abstractions without syntax errors. To bridge this, the transition roadmap requires a strict intermediate representation layer. Instead of asking the LLM to output raw logic or simple tags, we introduce a constrained generation step. Utilizing a specialized reinforcement learning approach to guide the structural graph generation can force the LLM to output strictly typed JSON or Pydantic models that map directly to your semantic types (Individuals $e$, Properties $\langle e, t \rangle$, and Quantifiers $\langle \langle e, t \rangle, t \rangle$). This ensures the pipeline feeding the Montague Engine never encounters malformed logic.

## 5. Cognitive Stratification and the Conversational Refinement Loop

To prevent "cognitive drift" and "data bloat," we stratify the knowledge graph into distinct architectural layers. This structure enforces a "bottom-up" development path, ensuring the archive remains an antifragile web of knowledge.

### The Three Layers of Evidence

1. **Layer 1: Data Description & Patterns**: The extensional "Factual Layer." This stores atomic entities and their basic relations (e.g., Anvil Corp, Summit Logistics, and the `ACQUIRED` relationship).
2. **Layer 2: Interpretation**: This layer applies the Logic of Ordinary English to resolve Intensional Contexts. The system must distinguish between *de dicto* (non-specific) and *de re* (specific) readings. For instance, "Anvil Corp is seeking a local acquirer" is interpreted *de dicto* (it seeks the property of being an acquirer) until a specific target is identified, at which point it shifts to *de re*. This prevents the premature creation of "ghost" nodes in Layer 1.
3. **Layer 3: Synthesis**: The "Autonomous Zettel" layer. Here, the engine identifies higher-order patterns—such as the "Key Local Acquirer" pattern—linking multiple Layer 1 facts into a new, synthesized insight node.

### The Conversational Refinement Loop

The Internal Zettelkasten acts as an autonomous synthesis engine that actively reflects on factual knowledge to build a network of higher-order insights, rather than functioning as a static structural strategy. Human interaction should primarily occur through a chat-driven interface, acting as a "manager" in the loop. When queried, the AI must first consult its internal Zettelkasten for synthesized insights rather than just returning a list of raw facts. The conversational interface allows the human to steer the AI's next synthesis cycle. Prompts asking for counter-examples should trigger the AI to initiate a new analysis and generate more nuanced insight nodes.

### Structural Layer Strategy for Neo4j

We implement a three-tier hierarchy in the graph schema:

* **Bottom Layer (Content Nodes)**: The raw $\lambda$-converted factual data.
* **Middle Layer (Structure Notes)**: Nodes that group related Content Nodes via `BELONGS_TO` links, acting as thematic clusters.
* **Top Layer (Main Structure Notes)**: Global indices and special tags (Double Hashes) that provide orientation across the entire cognitive architecture.

By prioritizing active interpretation over simple bookmarking, we bypass the "Collector's Fallacy," ensuring that every addition to the graph is the result of a deliberate "thinking path."

## 6. Transition Roadmap: Evolving the Local Acquirer MVP

Upgrading our Python/Neo4j stack requires evolving the extraction logic from "flat" parsing to "type-driven" derivation.

### Extraction Prompt Evolution

The current Extraction Prompt must be refactored to identify Syntactic Categories and Semantic Types. The LLM will no longer output simple triplets; it will identify:

* **Individuals ($e$)**: Proper names (e.g., "Anvil Corp").
* **Properties ($\langle e, t \rangle$)**: Common nouns (e.g., "Logistics Company").
* **Quantifiers ($\langle \langle e, t \rangle, t \rangle$)**: Determiners (e.g., "Every").

### Internally Motivated Type-Shifting

Following Yoad Winter's "Flexibility Principles," the system will implement Type-Shifting Operators to resolve mismatches between syntactic context and semantic type.

* **Atomic vs. Set-based Individuals**: The system will use "internally motivated shifting" to handle plurality. For example, "Simon and Garfunkel" can be treated as two atomic individuals or shifted into a single "set-based individual" (a songwriting team). This ensures that relationships like `WROTE_HITS` can be applied to the collective set rather than erroneously duplicating the relationship for each atomic member.

### Direct Compositionality on Demand (Case Study)

To handle long-distance dependencies like anaphora without the computational overhead of global Quantifier Raising, we implement Direct Compositionality on Demand.

* **Example**: "Anvil Corp acquired Summit Logistics. It is based in Anaconda."
* **Resolution**: The system decomposes this into local, compositional increments. The pronoun "it" is assigned a semantic value relative to the assignment function $g$. Through the derivational history of the first sentence, the engine resolves the anaphoric link to "Anvil Corp" as the most salient entity in the topical stack before the second sentence is fully processed.

## 7. Deployment: Exposing the Engine via Model Context Protocol (MCP)

The technical specification requires a deployment mechanism to make this an active, autonomous cognitive engine rather than a static script. It needs to be wrapped in a Model Context Protocol (MCP) server. The MCP server should be designed to expose the functionalities of both the Graph Database and the Internal Zettelkasten. These must be structured as callable tools and manageable resources. Implementing a dedicated coordinator to manage these connections will allow the AI's cognitive layers to act as independent MCP clients. This client-server decoupling empowers the AI to autonomously build, access, and refine its internal knowledge graph.

## 8. Robustness and Persistence: The "Perennial Garden" Model

A cognitive system meant to last decades must be treated as a "perennial garden"—a living system where the "soil" (the quality of connections) improves over time.

### Principles for Longevity Checklist

* **Portability**: All data is persisted in Markdown or Open Formats. We avoid "feature-bloated" tools in favor of stable, minimalistic environments like The Archive, which prioritizes plain text and high-speed search over "glowing graphs."
* **Connectivity**: We prioritize link-based structures over fixed hierarchical folders. Links reflect thinking paths; folders reflect static assumptions.
* **Scalability**: We promote "Bottom-Up" development. The system remains "boring" and stable, avoiding the "cognitive noise" of ephemeral plugins.

### Linguistic Side Effects (Continuations)

To manage complex control operators during synthesis, we implement Linguistic Side Effects via the "shift" and "reset" operators (as proposed by Chung-chieh Shan). We treat in-situ quantification (e.g., "Anvil Corp saw everyone") as a computational control operator. This allows the system to "reset" the evaluation context, handling the scope of the quantifier "everyone" as a side effect of the local verb-phrase processing.

By moving from pragmatic simulation to formalized understanding, this architecture ensures the knowledge base remains a stable, built-to-last library for peak cognition, moving AI from mere data retrieval to genuine autonomous synthesis.
